Cloud Storageには、４つの主要なストレージクラスがあります
１つ目はStandard Storageです
Standard Storageは利用頻度の高い、いわゆるホットデータに最適です
保存期間が短いデータにも適しています
２つ目はNearline Storageです
月に１回程度しかアクセスしない利用頻度の低いデータの保存に最適です
たとえばデータのバックアップ、ロングテールのマルチメディアコンテンツ、データのアーカイブなどです
３つ目はColdline Storageです
これも利用頻度の低いデータを低コストで保存できます
Nearline Storageと違い、Coldline Storageは90日に１回程度の利用頻度に向いています
４つ目はArchive Storageです
最も低コストのオプションで、データのアーカイブ、オンラインバックアップ、障害復旧に最適です
年に１回アクセスするかどうかというデータに向いています
データへのアクセスと処理にかかるコストが比較的高く、365日の最低保管期間があるからです
４つのクラスには違いがありますが、すべてのクラスに共通するいくつかの特性を抑えておきましょう
最小オブジェクトサイズの要件がない無制限ストレージです
世界中でアクセス、保存できます
低レイテンシで高耐久です
セキュリティ、ツール、APIを含め、エクスペリエンスが一貫しています
複数のリージョンに保存するデータは地理冗長化されます
つまり物理サーバーが地理的に分散したデータセンターに配置され、災害時のデータ保護や負荷分散による最適化が行われます
Cloud Storageに最低料金はなく、使用した分だけ料金が発生します
事前の容量プロビジョニングも不要です
セキュリティ面では、データはディスクへの書き込み前に常にサーバー側で暗号化されます
追加料金はかかりません
顧客デバイスとGoogleの間を移動するデータは、HTTPS/TLS（Transport Layer Security）でGoogleによりデフォルトで暗号化されます
どのストレージクラスを利用する場合もデータを格納する方法は複数あります
お客様の多くはgsutilを使ってシンプルにオンライン転送を行います
gsutilはCloud Storage操作用のCloud SDKコマンドです
Cloud Consoleでのドラッグ＆ドロップ操作でもデータを移動できます
ConsoleにはChromeからアクセスできます
テラバイトやペタバイト単位のデータはどうすればいいでしょう
Storage Transfer Serviceを使えば大容量のオンラインデータをCloud Storageにすばやく低コストで移せます
Cloud Storageへのバッチ転送のスケジュールを設定して管理できます
転送元は別のクラウドプロバイダや、別のCloud Storageリージョン HTTPSエンドポイントからも可能です
またTransfer Applianceという、Google Cloudがリースするラック搭載可能な大容量ストレージサーバーもあります
これを自社ネットワークに接続して、データを読み込み、この装置をGoogleに返送するとCloud Storageにデータがアップロードされます
１つの装置で最大１ペタバイトのデータを転送できます
Cloud StorageはGoogle Cloudのプロダクトやサービスと統合されているので、データをサービスに移す方法は他にもいつくもあります
たとえばBigQueryやCloud SQLとの間で、テーブルをインポート/エクスポートできます
またApp EngineログやFirestoreバックアップ、App Engineアプリで使われる画像などのオブジェクトも格納できます
Cloud Storageにはインスタンスの起動スクリプトやCompute Engineイメージ、Compute Engineアプリのオブジェクトも格納できます
