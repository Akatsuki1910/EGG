１つ目はCloud Storageです
開発者とIT組織に耐久性と可用性に優れたオブジェクトストレージを提供するサービスです
まずオブジェクトストレージとは、データをオブジェクトとして管理するデータストレージアーキテクチャで、ファイルやフォルダ階層として管理するファイルストレージや、ディスクチャンクとして管理するブロックストレージとは異なります
オブジェクトはパッケージ形式で保存され、中にはバイナリ形式のデータ本体とそれに関連するメタデータが入っています
メタデータには作成日、作成者、リソースタイプ、権限、グローバル一意識別子などが含まれます
これらの一意のキーはURL形式を取るため、オブジェクトストレージとウェブテクノロジーには親和性があります
一般にオブジェクトとして保存されるデータには、動画、画像、音声などがあります
Cloud StorageはGoogleのオブジェクトストレージプロダクトで、どんな容量のデータでも格納して必要に応じて何度でも取り出せます
フルマネージドのスケーラブルなサービスで、用途は多様です
たとえばウェブサイトコンテンツの提供、アーカイブと障害復旧のためのデータ保存、大容量データの直接ダウンロードでの配布などの用途があります
Cloud Storageの主な用途は、動画や写真などのオンラインコンテンツで、BLOB（Binary Large OBject）ストレージが必要になる場面です
データのバックアップやアーカイブ、中間結果や処理ワークフローの格納が目的です
Cloud Storageファイルはバケット化されます
バケットにはグローバル一意識別子と格納先の具体的な位置情報が必要です
格納先はレイテンシが最小になる場所が理想です
たとえばユーザーの大半がヨーロッパにいるならヨーロッパがよいでしょう
ヨーロッパ内の特定のリージョンかEUのマルチリージョンです
Cloud Storageのストレージオブジェクトは不変です
つまり編集されるのではなく変更のたびに新しいバージョンが作成されます
管理者は新バージョンで古いものを完全に上書きするか、バケット内でバージョン管理を有効にして変更を追跡できるようにするか選べます
バージョン管理を有効にすると、Google Storageは詳細な変更履歴つまり、バケット内のすべてのオブジェクトの上書きや削除の履歴を残します
バージョン管理が無効であると、古いバージョンが常に新しいバージョンで上書きされます
バージョン管理が有効だとオブジェクトのバージョンを一覧表示したり、オブジェクトを以前の状態に復元したり、バージョンを完全に削除したりできます
多くの場合、データオブジェクトには個人を特定できる情報が含まれます
セキュリティとプライバシーを確保するためには、保存データへのアクセスの制御が不可欠です
IAMロールと必要に応じてアクセス制御リストを使うことで、セキュリティのベストプラクティスに準拠できます
つまり各ユーザーのアクセス権限をそれぞれの業務に必要なリソースに限定できます
ユーザーのアクセス制御にはいくつかのオプションがあります
大半の目的はIAMで十分果たせます
プロジェクトからバケット、オブジェクトへとロールは継承されます
細かい制御が必要なら、アクセス制御リストを作成します
アクセス制御リストは２つの情報で構成されます
１つ目はスコープで、アクセスして操作できる人を定義します
ユーザーまたはグループを指定できます
２つ目は権限です
読み取りや書き込みなど、実行可能な操作を定義します
大量のデータの出し入れを繰り返していると、すぐにコストが高くなるので、Cloud Storageではライフサイクル管理ポリシーも使えます
たとえば365日以上経過したオブジェクトを削除したり、特定の日付より前に作成されたオブジェクトを削除するよう設定できます
バージョン管理が有効なバケットで直近３バージョンのみ保存することも可能です
こういった制御を通じて料金を必要最低限に抑えられます
