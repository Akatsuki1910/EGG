負荷分散と自動スケーリングについて説明します
Cloud Load Balancingを使用すると、負荷分散したコンピューティングリソースを単一または複数のリージョンに分散させ、高可用性要件を満たせます
単一のエニーキャストIPアドレスの背後にリソースをデプロイし、インテリジェントな自動スケーリング機能でリソースをスケーリングできます
Cloud Load Balancingにより、秒間クエリ数100万件以上に対応できるシステム上で可能な限りユーザーに近い場所でコンテンツを処理できます
Cloud Load Balancingは完全に分散されたSW定義型のマネージドサービスです
インスタンスベースでもデバイスベースでもないため、物理的な負荷分散インフラの管理が不要です
GCPに備わる多様なロードバランサは、グローバルとリージョンの２つのカテゴリに分類できます
グローバルLBには、HTTP(S)、SSLプロキシ、TCPプロキシのLBがあります
これらのLBは、Googleの接続拠点にグローバルに分散したSW定義の分散システムである、Googleフロントエンドを活用します
したがってグローバルLBの使用が適しているのは、ユーザーとインスタンスがグローバルに分散し、各ユーザーが同じアプリとコンテンツにアクセスする必要があり、かつアクセス手段として単一のエニーキャストIPアドレスを使用する場合です
リージョンLBには、内部LBとネットワークLBがあり、トラフィックを単一のGCPリージョン内のインスタンスに分散させます
内部LBはGCPのSW定義ネットワーク仮想化スタックであるAndromedaを使用し、ネットワークLBは大規模な分散SWシステムであるMaglevを使用します
HTTP(S)トラフィック向けの内部LBもあります
この第６のLBはプロキシベースのリージョンのレイヤ７LBであり、VPCネットワーク内のLBのリージョンでのみアクセスできるプライベート負荷分散IPアドレスの背後で、サービスを実行およびスケーリングできます
このモジュールでは、GCPで使用できるさまざまな種類のLBについて説明します
さらにこうした負荷分散構成で使用できるマネージドインスタンスグループと、その自動スケーリング構成についても説明します
このモジュールの２つのラボでは、多くの機能とサービスについて詳しく見ていきます
その後のまとめとして、ニーズに最適なGCPのLBを判断する方法を紹介します
では早速、マネージドインスタンスグループから説明します
